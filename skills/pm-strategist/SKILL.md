# PM Strategist — 自主产品策划工作流

## Purpose

产品策划的核心难题不是"想不出方案"，而是"想出的方案没有经过足够的质疑"。大多数产品经理在提出第一个方案后就开始爱上它，Critic 的声音被压了下去。这个 Skill 通过结构化的五阶段流水线和 GAN 式生成-批判循环，强制将"想方案"和"挑方案的刺"分离成两个独立过程，确保最终输出的方案是经过多轮对抗打磨的。

它的独特价值在于：
- **代码感知**：不只读需求，还读代码，方案建立在技术现实之上
- **对抗式推演**：不是生成一版就完事，而是 Generator 和 Critic 反复交锋
- **三版对比**：不给单一答案，给保守/平衡/激进三条路，让决策者有真正的选择
- **全自主运行**：从输入需求到输出完整文档，中间不需要人来回答问题

_为什么要借鉴 GAN？因为 Anthropic 的工程实践表明，让同一个系统既生成又评估会导致"自我原谅"——发现问题后又说服自己问题不大。分离 Generator 和 Critic 的角色，用独立的评分标准来逼出更好的结果，这个结构灵感来自 GAN，但落地形式是多角色 Agent 编排，不是机器学习训练。_

## Key Concepts

### 1. 五角色推演流水线

每个角色专注一类认知任务，产出结构化的中间产物（Artifact），传递给下一个角色。不跳跃、不混合。

| 角色 | 认知任务 | 输入 | 输出 |
|---|---|---|---|
| **Planner** | 需求拆解与问题定义 | 客户原始需求 | 问题定义、用户画像、假设树、成功标准 |
| **Researcher** | 代码与上下文检索 | 代码仓库 + 知识库 | 技术分析、约束、机会、信息缺口 |
| **Synthesizer** | 多方案生成 | Planner + Researcher 产物 | 三版差异化方案（保守/平衡/激进） |
| **Critic** | 批判评审与评分 | Synthesizer 的三版方案 | 四轴评分、质疑、修订要求 |
| **Validator** | 最终一致性校验 | 通过批判的推荐方案 | 校验报告、置信度评估 |

_为什么五个而不是三个或七个？因为少于五个会丢失关键认知步骤（比如没有独立的 Critic 就没有对抗性），多于五个会引入不必要的编排复杂度。五个是信息处理的最小完备集。_

### 2. GAN 式生成-批判循环

这不是机器学习意义上的 GAN 训练，而是借鉴其核心直觉：**让"造东西的"和"挑刺的"永远是两个独立的角色**。

```
Synthesizer 生成三版方案
       ↓
Critic 用四轴评分框架逐项打分
       ↓
   加权总分 >= 7.0？──→ 是 ──→ 最高分方案进入 Validator
       ↓ 否
Refiner 只修改 Critic 标记为失败的维度
       ↓
回到 Critic 重新评分（最多 4 轮）
       ↓
4 轮后仍未通过 ──→ 取最高分方案 + 附加"已知局限"声明
```

**四轴评分框架**：

| 评分轴 | 权重 | 评估内容 |
|---|---|---|
| 需求匹配度 | 30% | 方案是否真的解决了客户声明和未声明的需求？ |
| 技术可行性 | 25% | 现有代码库和团队能否现实地构建这个？ |
| 商业价值 | 25% | 这个方案是否推动了有意义的业务指标？ |
| 风险可控度 | 20% | 风险是否被识别、可缓解、可恢复？ |

评分标准：每轴 1-10 分。加权总分 >= 7.0 为 PASS，< 7.0 为 REVISE。

### 3. Sprint Contract（置信度协商）

参考 Anthropic 原文的 Sprint Contract 概念。在 Critic 首轮评审前，先定义"什么算好方案"的验收标准。这避免了 Critic 事后移动标尺。

Sprint Contract 示例：
```
本次策划的验收标准：
1. 方案必须在现有技术栈上可实现（不引入全新语言/框架）
2. MVP 阶段 <=3 个月上线
3. 至少解决客户声明的核心需求（80% 覆盖）
4. 风险清单中不得有"灾难性且无缓解方案"的风险
```

### 4. 三版方案差异化原则

| 维度 | 保守版 | 平衡版 | 激进版 |
|---|---|---|---|
| 哲学 | 最小改动，最大复用 | 创新与稳定的平衡 | 最大用户价值，接受更高风险 |
| 技术路径 | 扩展现有系统 | 适度重构 + 新组件 | 显著新架构 |
| 首次价值交付 | 最快 | 中等 | 最慢 |
| 范围 | 窄 MVP | 核心功能 + 差异化 | 完整愿景 |
| 风险 | 低 | 中 | 高 |
| 资源 | 最少 | 适中 | 显著 |

**铁律：三版必须在技术路径、时间线、资源需求上有实质差异。禁止只是措辞不同。**

### 5. 自主 Skill 调度

Agent 不需要人告诉它该用哪个 Skill。它根据当前阶段和上下文信号自动决策：

| 阶段 | 信号 | 自动调用的 Skill |
|---|---|---|
| Planner | 所有场景 | `problem-statement`, `jobs-to-be-done` |
| Planner | 提到竞品 | `company-research` |
| Planner | 涉及市场力量 | `pestel-analysis` |
| Researcher | 提供了代码仓库 | `code-explorer` SubAgent |
| Researcher | 有知识库 | `product-knowledge-base` |
| Synthesizer | 所有场景 | `requirement-generator`, `roadmap-planning` |
| Synthesizer | 面向终端用户 | `proto-persona`, `customer-journey-map` |
| Synthesizer | 需要 ROI 分析 | `feature-investment-advisor` |
| Synthesizer | 需要市场规模 | `tam-sam-som-calculator` |
| Critic | 所有场景 | `multi-perspective-evaluation` |
| Synthesizer | 优先级冲突 | `prioritization-advisor` |

## Application

### 输入要求

用户只需提供：

1. **必需**：客户需求描述（自然语言，任意详细程度）
2. **强烈推荐**：目标代码仓库路径或链接
3. **可选**：产品背景文档、已知约束条件、优先级偏好

示例输入：
```
客户需求：我们的电商平台需要一个智能推荐系统，用户反馈现在的"猜你喜欢"
完全不准，点击率只有 2%，行业平均是 8%。

代码仓库：/path/to/ecommerce-platform

已知约束：
- 团队只有 3 个后端 + 2 个前端，没有 ML 工程师
- Q3 必须上线，预算不超过 20 万/月
- 不能换主数据库（MySQL 8.0）
```

### Phase 1: Planner — 需求拆解与问题定义

**你要做什么**：把客户的原始需求翻译成结构化的问题空间。

**操作步骤**：

1. **提取核心问题**：调用 `problem-statement` skill，产出格式化的问题陈述
   - 谁被阻塞？做什么？为什么重要？不解决会怎样？

2. **挖掘用户动机**：调用 `jobs-to-be-done` skill
   - 功能性任务：用户想完成什么？
   - 情感性任务：用户想感受什么？
   - 社会性任务：用户想被怎样看待？

3. **构建假设树**：
   ```
   H1: 推荐不准是因为没有用户行为数据 → 证据: [待验证] → 如果错了: 加数据也不会改善
   H2: 推荐不准是因为算法太简单 → 证据: [看代码] → 如果错了: 换算法是浪费
   H3: 用户根本不看推荐区域 → 证据: [需要埋点] → 如果错了: 即使推荐准也没效果
   ```

4. **定义成功标准和范围边界**

**Phase 1 产出检查清单**：
- [ ] 问题陈述清晰，不包含解决方案假设
- [ ] 至少识别了 2 个目标用户群
- [ ] 假设树中每个假设都标注了证据来源和失败条件
- [ ] 成功指标有基线值和目标值
- [ ] 明确列出了"不解决什么"

---

### Phase 2: Researcher — 代码与上下文检索

**你要做什么**：全面理解技术现实和产品上下文，让后续方案建立在事实之上。

**操作步骤**：

1. **代码仓库分析**（通过 `code-explorer` SubAgent）：
   - 读取目录树，识别模块划分
   - 查看 README 和文档，理解项目定位
   - 分析关键接口定义和数据模型
   - 检查配置文件，了解基础设施和依赖
   - 浏览测试文件，推断哪些功能被重视

2. **知识库检索**（通过 MCP 工具）：
   ```
   search_knowledge(query="[需求相关关键词]", top_k=5)
   get_templates(category="requirement")
   get_progress_board()
   ```

3. **产出能力-机会映射表**：
   | 现有能力 | 产品机会 | 技术成本 | 约束 |
   |---|---|---|---|
   | 用户行为日志 | 可用于推荐训练 | 低 | 日志格式需标准化 |
   | MySQL 全文搜索 | 基础推荐 | 低 | 性能瓶颈在大数据量 |

4. **标注信息缺口**：
   - 明确列出"我不知道但影响方案质量的信息"
   - 对每个缺口标注：可以继续推进 vs 这是阻塞性的

**Phase 2 产出检查清单**：
- [ ] 技术栈和架构模式已识别
- [ ] 关键模块及其职责已列出
- [ ] 数据模型和 API 表面已分析
- [ ] 至少检索了 3 条知识库相关文档
- [ ] 能力-机会映射表已完成
- [ ] 信息缺口已标注置信度

---

### Phase 3: Synthesizer — 三版方案生成

**你要做什么**：基于 Phase 1 和 Phase 2 的产物，生成三版差异化的产品方案。

**操作步骤**：

1. **确定三版方案的差异化方向**：
   - 保守版：在现有能力上做最小扩展
   - 平衡版：引入适度创新，兼顾可控性
   - 激进版：追求最大用户价值，接受更高风险和成本

2. **对每版方案**，依次完成：
   - 调用 `requirement-generator` 生成结构化 PRD 骨架
   - 调用 `roadmap-planning` 规划实施阶段
   - 调用 `proto-persona`（如需要）勾画目标用户画像
   - 写明 Trade-off 分析：选择这条路意味着放弃什么
   - 预填"为什么这版可能是错的"——在 Critic 之前先自我挑战

3. **产出对比表**：

   | 维度 | 保守版 | 平衡版 | 激进版 |
   |---|---|---|---|
   | 上线时间 | | | |
   | 用户价值 | | | |
   | 技术债务 | | | |
   | 风险等级 | | | |
   | 资源需求 | | | |
   | 长期可扩展性 | | | |

**Phase 3 产出检查清单**：
- [ ] 三版方案在技术路径上有实质差异（不只是措辞）
- [ ] 每版有完整的 PRD 骨架（问题/方案/指标/风险/路线图）
- [ ] Trade-off 分析诚实标注了每版的牺牲
- [ ] "为什么这版可能是错的"不是空话
- [ ] 对比表可以让非技术人员快速理解差异

---

### Phase 4: Critic — GAN 批判器

**你要做什么**：独立、严格地评审每版方案。这是整个流水线最关键的阶段。

**操作步骤**：

1. **建立 Sprint Contract**（仅首轮）：
   - 定义"什么算好方案"的验收标准
   - 标准必须具体、可评分，不能是"方案要好"

2. **调用 `multi-perspective-evaluation`**：
   - 五维度（怀疑论者/用户拥护者/执行者/远见者/系统思考者）逐一评审

3. **执行四轴评分**：
   ```
   | 评分轴 | 方案A得分 | 方案B得分 | 方案C得分 |
   |---|---|---|---|
   | 需求匹配度 (30%) | | | |
   | 技术可行性 (25%) | | | |
   | 商业价值 (25%) | | | |
   | 风险可控度 (20%) | | | |
   | 加权总分 | | | |
   | 判定 | PASS/REVISE | PASS/REVISE | PASS/REVISE |
   ```

4. **对每个 REVISE 判定**：
   - 列出必须修改的具体条目（不是泛泛的"需要改进"）
   - 指明修改方向和参考依据
   - 标注"不要动"的部分（Strengths to Preserve）

5. **迭代循环**（如需要）：
   - Refiner 只修改 Critic 标记的失败维度
   - 回到 Critic 重新评分
   - 最多 4 轮

**Critic 的禁止行为**：
- ❌ 使用"总体还好"、"基本可以"等模糊评价
- ❌ 发现问题后又自行"原谅"（Anthropic 原文明确指出这是默认 AI 的坏习惯）
- ❌ 对三版方案给出几乎相同的分数（如果分数差异 < 1.0，说明评审不够深入）
- ❌ 跳过任何一个评分轴

**Phase 4 产出检查清单**：
- [ ] Sprint Contract 已建立（首轮）
- [ ] 五维度批判报告已完成
- [ ] 四轴评分已填写，每轴有具体理由
- [ ] REVISE 方案有明确的修改清单
- [ ] 三版方案的分数有合理差异（> 1.0）

---

### Phase 5: Validator — 最终校验

**你要做什么**：确认推荐方案在逻辑上自洽、与代码现实一致、可以交付给利益相关者。

**操作步骤**：

1. **执行校验清单**：
   | 校验项 | 状态 | 备注 |
   |---|---|---|
   | 问题定义与原始需求一致 | ✅/❌ | |
   | 方案在现有代码库上可行 | ✅/❌ | |
   | 成功指标可用现有或计划中的工具测量 | ✅/❌ | |
   | 风险缓解措施具体可执行 | ✅/❌ | |
   | 路线图时间线符合代码复杂度 | ✅/❌ | |
   | PRD 各部分无内部矛盾 | ✅/❌ | |
   | 范围边界清晰 | ✅/❌ | |
   | 自我批判诚实 | ✅/❌ | |

2. **置信度评估**：
   - 整体置信度：高/中/低
   - 最大不确定性来源
   - 什么信息能提高置信度

3. **判定**：APPROVED → 输出最终文档 / NEEDS REVISION → 返回 Synthesizer

**Phase 5 产出检查清单**：
- [ ] 8 项校验全部完成
- [ ] 校验不通过的项有修正说明
- [ ] 置信度评估诚实标注了最大不确定性
- [ ] 最终判定明确

---

### 最终输出

当五阶段完成后，汇编为一份完整的产品策略文档（见 `agents/pm-strategist.md` 中的 Final Deliverables 模板），包含：

1. 执行摘要
2. 问题定义
3. 技术与产品上下文
4. 三版方案对比
5. 批判审查记录
6. 推荐方案完整 PRD
7. 实施路线图
8. 风险清单
9. 成功指标与测量方案
10. 已知局限与开放问题
11. 附录：迭代日志

## Examples

### 完整示例：电商智能推荐系统

**输入**：
```
客户需求：电商平台的"猜你喜欢"点击率只有 2%（行业平均 8%），
用户反馈推荐完全不准。团队 3 后端 + 2 前端，无 ML 工程师，
Q3 前必须上线，月预算 <=20 万。代码仓库已提供。
```

**Phase 1 Planner 示例产出**（节选）：

```markdown
### 问题定义
电商平台的推荐系统效果严重低于行业标准（2% vs 8% 点击率），导致用户发现
商品的能力受限，可能影响转化率和客单价。用户的核心困扰不是"没有推荐"，
而是"推荐了但不相关"。

### 假设树
- H1: 推荐不准是因为用户行为数据采集不足
  → 证据: 需看代码中的埋点覆盖率
  → 如果错了: 即使补全数据，模型仍可能不够好

- H2: 推荐算法过于简单（如仅基于热门排序）
  → 证据: 需看推荐模块代码
  → 如果错了: 换复杂算法成本高但收益可能有限

- H3: 推荐展示位置/UI 不够突出，用户看不到
  → 证据: 需要前端埋点数据
  → 如果错了: 即使位置好，内容不相关也无用

### 成功标准
| 指标 | 当前 | 目标 | 测量方法 |
|---|---|---|---|
| 推荐点击率 | 2% | >=6% (Q3) | GA 事件追踪 |
| 推荐转化率 | 0.3% | >=1.5% | 归因分析 |
| 推荐覆盖率 | 未知 | >=80%商品 | 后端日志 |
```

**Phase 4 Critic 示例产出**（节选）：

```markdown
### Critic Review: 方案B（平衡版：协同过滤 + 内容推荐混合）
```

## Common Pitfalls

### 反模式 1: Critic 走过场

**症状**: Critic 给三版方案都打了 7-8 分，评语都是"基本可行，建议进一步完善"。

**为什么危险**: 这说明 Critic 没有真正做批判，而是在做"鼓励式评审"。Anthropic 的工程实践明确指出：默认的 AI 是糟糕的 QA——它倾向于发现问题后又原谅自己。

**怎么修**: Critic 的 Prompt 中有铁律——禁止使用模糊评价，每轴必须给具体扣分理由，三版分数差异必须 > 1.0。如果你发现 Critic 在走过场，手动要求"重新评审，这次请认真找问题"。

### 反模式 2: 三版方案实质相同

**症状**: 保守版是"用简单算法"，平衡版是"用中等算法"，激进版是"用复杂算法"——本质上是同一个方案的三种参数配置。

**为什么危险**: 这不是真正的选择。真正的三版应该在技术路径、产品策略、甚至解决问题的角度上都不同。

**怎么修**: 强制每版至少有一个"如果选了这条路就不能选那条路"的排他性决策。比如保守版"不引入ML，用规则引擎"；平衡版"引入轻量ML但不换架构"；激进版"全面重建推荐服务，接入第三方ML平台"。

### 反模式 3: 忽略代码现实

**症状**: 方案写得很漂亮，但完全没有参考 Researcher 阶段的代码分析。提出的技术路径和现有代码架构不兼容。

**为什么危险**: 这让整个策划变成了"空中楼阁"。PM 的方案如果与代码现实脱节，工程师会直接否决，信任也随之崩溃。

**怎么修**: Validator 阶段的校验清单中有"方案在现有代码库上可行"这一项。如果这项不通过，必须回到 Synthesizer 修改方案，而不是假装问题不存在。

### 反模式 4: 完美主义陷阱

**症状**: GAN 循环跑了 4 轮还不满意，想继续跑第 5、6、7 轮。

**为什么危险**: 迭代的边际收益递减很快。第 3 轮和第 4 轮的改进通常是细节级别的，但消耗的时间和 token 是线性增长的。

**怎么修**: 严格遵守 4 轮上限。如果 4 轮后仍未通过，说明问题可能不在方案质量，而在输入信息不足。此时正确的做法是输出"最佳可用方案 + 已知局限"，让人类决策者判断。

### 反模式 5: 跳过 Planner 直接写方案

**症状**: 觉得"客户需求很清楚了，不需要再拆解"，直接跳到 Synthesizer。

**为什么危险**: 客户说的和客户需要的几乎永远不完全一致。跳过 Planner 意味着你接受了客户的问题框架，而不是重新定义问题。很多产品失败不是方案不好，而是解决了错误的问题。

**怎么修**: Planner 是必执行阶段，不能跳过。即使需求看起来很清楚，Planner 至少应该完成假设树和范围界定。
