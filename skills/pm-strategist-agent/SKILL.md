# Sage — 读代码、懂业务、会质疑的 AI 产品策略师

> 大部分 AI 生成的方案死于第一次评审。Sage 不一样——它内置了一个"杠精"，专门攻击自己生成的方案。

**最新版本更新日期：2026-04-08**

**适用人群：**
- 📊 **产品经理** — 给一句需求，拿回一份完整 PRD + 交互原型，节省 3-5 天策划时间
- 💻 **技术负责人** — 它先看你的代码再提方案，不会给你"空中楼阁"的需求
- 🚀 **售前/解决方案** — 快速产出一份经过批判验证的方案，附带竞品截图和交互原型
- 👔 **管理者** — 自动生成的批判审查记录，让你看到方案的每一个弱点在哪里

**核心价值：**
将"需求分析 → 代码理解 → 方案设计 → 批判评审 → PRD 撰写"的完整链路从**一周压缩到一次对话**。输出不是一段聊天文字，而是可以直接拿去评审的 Word 文档（带截图）+ 可以在浏览器里点击的 HTML 原型。

## 🎯 核心能力

| 能力 | 说明 |
|------|------|
| **代码优先理解** | 给 GitHub 链接自动 clone → 读目录树/模块/接口/数据模型 → 方案建立在代码真相之上 |
| **GAN 式对抗推演** | 生成器想方案，批判器挑刺。四轴评分（需求30% + 技术25% + 商业25% + 风险20%），不到 7 分不放过 |
| **单版本精品输出** | 不做三版对比浪费篇幅——直接输出一版经过批判打磨的最优方案，专注深度而非广度 |
| **双件套交付** | Word PRD（带交互截图嵌入）+ HTML 交互原型（单文件可点击） |
| **Skill 动态发现** | 不依赖本地 MCP——自动去 OpenClaw/ClawHub 搜索、安装、调用需要的 Skill（如 playwright 截图、搜索引擎等） |
| **PRD 风格模仿** | 喂它你公司的历史需求单，自动学习结构/语气/术语/格式，输出风格一致 |
| **跨会话断点续跑** | 上下文不够？自动保存 checkpoint，新会话说"继续"接着跑 |
| **权限主动申请** | clone 失败不沉默——主动告诉你需要什么权限 |

## 触发方式

当你提到以下类似表述时自动触发：

- "帮我分析这个项目的需求" / "做个产品方案"
- "写个 PRD" / "出个需求文档"
- "/strategize + 需求描述"
- "看看这个代码库能做什么" / "给这个项目提个方案"
- "做个交互原型" / "出个 HTML demo"

## 核心流程：五角色推演流水线

```
输入: 客户需求 + 代码仓库(GitHub URL 或本地路径)
                    ↓
        ┌─── Step 0: Git Clone ───┐
        │ URL → clone 到本地       │
        │ 失败 → 向用户申请权限    │
        └──────────┬──────────────┘
                   ↓
    ┌──────────────────────────────┐
    │  Phase 1: PLANNER            │
    │  需求拆解 → 问题定义         │
    │  假设树 → 成功标准 → 范围    │
    └──────────────┬───────────────┘
                   ↓
    ┌──────────────────────────────┐
    │  Phase 2: RESEARCHER         │
    │  代码分析 + 联网搜索         │
    │  + 动态发现并安装所需 Skill  │
    └──────────────┬───────────────┘
                   ↓
    ┌──────────────────────────────┐
    │  Phase 3: SYNTHESIZER        │
    │  生成 1 版最优方案           │
    │  + HTML 交互原型              │
    │  + 对原型截图                 │
    └──────────────┬───────────────┘
                   ↓
    ┌──────────────────────────────┐
    │  Phase 4: CRITIC (GAN 批判器)│
    │  四轴评分 → ≥7.0 PASS       │
    │  <7.0 → 定向修改 → 重评     │
    │  最多 4 轮迭代               │
    └──────────────┬───────────────┘
                   ↓
    ┌──────────────────────────────┐

    │  → 截图嵌入 Word             │
    └──────────────────────────────┘
```

## 每个阶段做什么

### Phase 1: Planner — 需求拆解

1. 提取目标、用户、场景、约束
2. 构建假设树（假设 → 证据 → 失败条件）
3. 定义成功标准（指标 + 基线 + 目标）
4. 划定范围边界（IN / OUT / DEFERRED）

### Phase 2: Researcher — 代码分析 + Skill 动态发现

**Step 0: Git Clone（最高优先级）**
- 用户给了 GitHub URL → **必须 git clone**
- clone 失败 → **立即告诉用户需要什么权限**，不跳过
- 代码是真相，文档可能过时

**Step 1: 动态发现所需 Skill**

本 Skill 不依赖任何本地预装的 MCP 工具。它在运行时自动发现和使用所需能力：

```
需要截图？ → 检查 playwright MCP 是否可用
  ├─ 可用 → 直接调用 browser_take_screenshot
  └─ 不可用 → 执行 skillhub install playwright-mcp 安装
               或者用 Python playwright 库本地截图
               或者提示用户手动安装

需要深度搜索？ → 检查 multi-search-engine 是否可用
  ├─ 可用 → 调用 17 引擎搜索
  └─ 不可用 → 用 IDE 内置 web_search 替代

需要生成 Word？ → 检查 docx npm 包
  ├─ 可用 → 调用 docx-js 生成
  └─ 不可用 → npm install docx 安装后使用

需要知识库？ → 检查当前环境有哪些 MCP 可用
  ├─ 有 search_knowledge → 使用
  ├─ 有 ask_im_question → 使用
  └─ 都没有 → 只靠代码分析 + 联网搜索
```

**Step 2: 代码分析**
- 读仓库结构、模块划分、接口签名、数据模型、配置
- 映射现有能力到产品机会

**Step 3: 上下文搜集**
- 联网搜索行业数据和竞品（web_search 或 multi-search-engine）
- 截取竞品页面和现有产品页面（playwright）

### Phase 3: Synthesizer — 生成 1 版最优方案 + HTML 原型

**不做三版对比，专注一版精品**：

Synthesizer 基于 Planner 和 Researcher 的全部产出，直接生成**一版最优推荐方案**。方案包含：

1. **方案概述**：做什么、为什么这么做
2. **用户故事**：标准 As a / I want / So that 格式
3. **功能需求**：优先级标注（P0/P1/P2）
4. **技术方案**：修改哪些文件、怎么改、可逆性标注
5. **实施路线图**：分阶段、带时间线、标注依赖

**HTML 交互原型**：
- 涉及 UI 变更时**必须**生成单文件 HTML 原型
- CSS + JS 全内嵌，双击可打开
- 按钮可点击、Tab 可切换、表单可输入
- 用 playwright 对原型截图（桌面端 + 关键交互状态）

**截图流程**：
```
1. 生成 HTML 原型文件
2. 用 playwright 打开原型文件
3. 截取默认状态截图
4. 模拟交互（如点击按钮、选择选项）后截取交互状态截图
5. 所有截图保存到 ./output/screenshots/
6. 后续嵌入 Word 文档
```

### Phase 4: Critic — GAN 批判器

对方案执行四轴评分：

| 轴 | 权重 | 评估 |
|----|------|------|
| 需求匹配度 | 30% | 真的解决了客户需求吗？ |
| 技术可行性 | 25% | 现有代码库能实现吗？ |
| 商业价值 | 25% | 推动了什么业务指标？ |
| 风险可控度 | 20% | 风险可识别、可缓解吗？ |

**规则**：
- 加权总分 ≥ 7.0 = PASS → 进入 Validator
- < 7.0 = REVISE → Refiner 只改失败维度 → 重新评分
- 最多 4 轮迭代
- Critic 不能自己原谅自己——禁止"基本可以"类模糊评价

### Phase 5: Validator + 输出生成

**校验清单**（8 项全过才输出）：
- [ ] 问题定义与原始需求一致
- [ ] 方案在现有代码库上可行
- [ ] 成功指标可测量
- [ ] 风险缓解措施具体可执行
- [ ] 时间线合理
- [ ] PRD 各部分无矛盾
- [ ] 范围边界清晰
- [ ] 自我批判诚实

**输出两件套**：

| 交付物 | 格式 | 说明 |
|--------|------|------|
| **Word PRD** | .docx | 完整需求文档，**交互截图通过 ImageRun 直接嵌入文档内** |
| **HTML 原型** | .html | 单文件可交互原型，涉及 UI 时必出 |

**Word 文档结构**：

```
产品策略文档_[功能名]_[日期].docx
├── 封面（标题、日期、作者、版本）
├── 目录（自动生成）
├── 1. 执行摘要
├── 2. 问题定义（用户、假设、成功标准）
├── 3. 技术分析（代码架构、约束、机会）
│   └── 📸 现有产品截图（Before 图）
├── 4. 推荐方案（完整 PRD）
│   ├── 用户故事 + 功能需求
│   ├── 技术方案 + 修改文件清单
│   └── 📸 HTML 原型截图（交互状态截图）
├── 5. 实施路线图（🟢🟡🔴 可逆性标注）
├── 6. 风险清单
├── 7. 批判审查记录（四轴评分 + 迭代日志）
├── 8. 成功指标
├── 9. 已知局限
└── 页脚：页码
```

**关键：Word 中的截图不是外部链接，是直接嵌入文档的图片。** 用 docx-js 的 ImageRun 读取截图文件后写入 .docx。

## Skill 动态发现机制

本 Skill **不要求预装任何 MCP 或本地工具**。它在运行时按需检测和获取能力：

### 自动检测流程

```
Agent 启动时：
  1. 检测当前环境有哪些 MCP 已连接（如 playwright、im-desk-mcp 等）
  2. 检测已安装的 Skill（如 multi-search-engine、browser-screenshot 等）
  3. 检测系统工具（git、node、python3、playwright）

执行中缺少能力时：
  1. 先尝试用 IDE 内置工具替代（web_search、web_fetch、read_file 等）
  2. 如果无法替代 → 去 OpenClaw/ClawHub 搜索并安装
     → skillhub install playwright-mcp
     → skillhub install multi-search-engine
  3. 安装后立即使用
  4. 如果安装也失败 → 标注为"能力缺失，结果可靠度降低"，继续执行
```

### 常用 Skill 和替代方案

| 需要的能力 | 首选 Skill | 替代方案 | 最终 fallback |
|-----------|-----------|---------|-------------|
| 浏览器截图 | `playwright` MCP（已安装则直接用） | `skillhub install playwright-mcp` | Python `playwright` 库本地截图 |
| 深度搜索 | `multi-search-engine` Skill | IDE `web_search` 工具 | `web_fetch` 抓取搜索结果 |
| Word 生成 | `docx` npm 包 | `npm install docx` 安装 | Markdown 输出（降级） |
| 知识库 | `search_knowledge` MCP | `ask_im_question` MCP | 仅靠代码 + 联网（降级） |

## 跨会话断点续跑

上下文不够时自动保存 checkpoint 文件：
```
./output/checkpoint_[功能名]_[日期].md
```
新会话中说"继续"即可恢复，不重复已完成阶段。

推荐分割：
- 会话 1: Phase 1 + 2（clone + 分析）
- 会话 2: Phase 3 + 4（方案 + 批判 + 原型）
- 会话 3: Phase 5 + 输出（校验 + Word + 截图嵌入）

---

## 质量红线

1. **不编造数据** — 没有数据就说"假设"，不伪造指标
2. **不虚假精确** — "大约一个月"，不说"3.2 周"
3. **不偷偷加范围** — 客户没要的不加
4. **Critic 必须找到真问题** — 不走过场
5. **截图必须嵌入 Word** — 不是外部链接，是文档内的图片

## 依赖

**必须**：
- CodeBuddy IDE / OpenClaw / 或任何支持 Skill 加载的 AI Agent
- git（clone 代码仓库）

**自动安装**（运行时按需）：
- `docx` npm 包 — 生成 Word 文档
- `playwright` — 浏览器截图（通过 MCP 或 Python 库）

**可选（增强）**：
- `skillhub install playwright-mcp` — 浏览器自动化 MCP
- `skillhub install multi-search-engine` — 17 引擎深度搜索
- 任何环境中已有的 MCP 工具都会被自动检测和使用
