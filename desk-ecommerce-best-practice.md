# 腾讯云智能客服 Desk · 电商平台接入最佳实践

---

## 一、适用场景

在电商平台竞争日益激烈的背景下，平台不仅需要提升买家咨询体验，也需要帮助入驻商家更高效地承接咨询、沉淀线索、处理订单与售后。尤其是在多商户经营模式下，如何让每个商家都拥有独立的服务能力，同时又能由平台统一管理、统一部署、统一运营，已经成为平台提升服务效率和商家经营效率的重要课题。

### 场景一：多商家商城场景

适用于平台内存在多个入驻商家、多个店铺、多个咨询入口的经营模式。平台希望统一建设服务底座，同时保证每个商家可独立运营自身服务能力，形成「平台统一管理、商家独立接待、买家体验清晰」的服务体系。

#### 1.1 零售商城

适用于服饰、美妆、家居、3C、玩具等以终端消费者为主的零售购物平台。

| 维度 | 说明 |
|------|------|
| **平台侧核心诉求** | 不是替每个店铺重做一套客服系统，而是提供一套统一底座，让商家入驻后即可快速拥有自己的服务能力。 |
| **真实业务场景** | 不同店铺的品牌定位、商品内容和服务风格差异很大，平台既要保证每个店铺能"有自己的服务风格"，又不能让平台运营团队逐店手工建设。 |
| **典型平台痛点** | **① 商家接入效率低**：新商家入驻后客服能力需要从 0 搭建，开店速度慢，影响招商节奏<br>**② 中小商家能力不足**：没有运营团队建设知识库和接待流程，大量中小商家的客服体验拉低平台整体水平<br>**③ 体验参差不齐**：各店铺话术、欢迎语、服务流程各行其是，买家在不同店铺感知割裂<br>**④ 跨店接待无统一入口**：同一客服服务多店但后台需逐店切换，人力重复消耗 |

#### 1.2 批发商城

适用于 B 端批发、分销、供应链等以企业买家为主的采购型购物平台。

| 维度 | 说明 |
|------|------|
| **平台侧核心诉求** | 让平台上的高价值询盘不被普通聊天淹没，让"咨询流量"能够沉淀成"可跟进商机"。 |
| **真实业务场景** | 买家不是来问"这件商品好不好看"，而是来问"1000 件什么价格""能不能贴牌""有没有长期供货能力"，每一条对话背后都可能对应一笔大订单。 |
| **典型平台痛点** | **① 高价值询盘承接不稳定**：商家客服靠经验判断要不要追问，线索流失，平台流量价值无法兑现<br>**② 询盘信息零散**：对话里散落商品、数量、公司、联系方式，商家跟进成本高<br>**③ 推广流量缺少转化链路**：广告页、活动页引流进来后没有标准承接，推广 ROI 难评估<br>**④ 询盘能力无法标准化输出**：平台想赋能商家，但没有可复制的采集流程，不同商家效果差异大 |

> *[示意图：批发商城 · AI 引导结构化询盘，线索回传业务后台]*

#### 1.3 店铺多入口差异化接待

同一个店铺，并不只有一个咨询场景。买家可能从店铺首页进来，也可能从商品详情页点进来，可能正在看活动专题，也可能是在订单页遇到了麻烦。入口不同，买家的心态和诉求完全不同。

| 维度 | 说明 |
|------|------|
| **平台侧核心诉求** | 不是简单给商家一个统一咨询入口，而是让平台上的不同流量入口，能够承接不同类型的问题与转化目标。 |
| **真实业务场景** | 店铺页买家还在判断"这家店值不值得逛"；商品页买家更关心参数、库存、优惠和购买建议；活动页买家最在意活动规则、优惠叠加和权益领取；订单页买家诉求通常已经非常明确，就是履约与售后。 |
| **典型平台痛点** | **① 欢迎语无差异**：所有入口共用一套欢迎语和快捷问题，买家需反复输入自己的诉求，承接效率低<br>**② 活动流量堆在人工侧**：活动规则问题无法自动化承接，大促期商家人工客服爆单<br>**③ 入口间逻辑混杂**：商品页、活动页、订单页共用一条对话流，转化型咨询与售后问题互相干扰 |

**不同入口 × 买家心智**

| 入口 | 买家心智 |
|------|---------|
| 店铺首页 | 了解品牌、主营品类、合作能力 |
| 商品详情页 | 了解规格参数、库存、搭配推荐 |
| 活动专题页 | 了解活动规则、优惠叠加、领取权益 |
| 订单详情页 | 查物流、催发货、改地址、退换货 |

### 场景二：平台级问题统一承接场景

在多商户购物平台中，买家除了咨询商家，还有大量问题需要找「平台」——这些问题都不是单个商家能处理的。如果分散在商家侧自行回答，容易出现规则口径不一致、投诉入口找不到、账号支付问题升级成舆情的情况。本场景在购物首页、个人中心、帮助中心、消息中心等平台级入口部署官方 AI 助手，专门承接这些平台级问题，与商家客服形成清晰职责边界。

| 维度 | 说明 |
|------|------|
| **平台侧核心诉求** | 为平台级问题提供统一、可信、可追溯的承接路径：统一规则解释出口、统一账号与支付求助通道、统一投诉举报受理机制。 |
| **真实业务场景** | 大促期买家对规则解释需求集中爆发；账号冻结、支付失败等问题买家天然直接找平台；投诉虚假发货、跨店纠纷时买家希望看到「提交给谁、什么时候处理、结果什么时候出」。 |

**三类平台级问题 × 典型痛点：**

| 问题类别 | 典型业务场景 | 不接入时的平台痛点 |
|---------|------------|-----------------|
| **平台规则咨询** | 大促满减规则、保价规则、退款规则、发票与权益 | 规则解释分散在商家侧，口径不一；买家因看不懂规则犹豫下单，影响转化；商家被大量平台规则问题占用服务资源 |
| **账号安全与支付纠纷** | 账号冻结、实名认证异常、支付失败、重复扣款、退款未到账 | 买家情绪升级快、对时效要求高；商家无权处理，平台若无统一入口体验最差；账号和资金问题直接影响平台信任感 |
| **投诉举报与跨商家调解** | 虚假发货、假冒商品、虚假宣传、跨店纠纷、仲裁申诉 | 投诉入口分散，买家不知道找谁；商家与平台责任边界不清；缺少统一受理—跟进—反馈机制，未承接的投诉易演变为舆情 |

> *[示意图：平台官方客服号 · 独立于所有商家客服]*

### 场景三：会员差异化服务场景

钻石会员、黑卡会员、月消费过万的高价值买家和普通用户采用同一条服务通道，是最容易流失高价值客户的方式。本场景贯穿多商家商城与平台客服两大场景，让重要的人走重要的通道。

| 维度 | 说明 |
|------|------|
| **平台侧核心诉求** | 不是简单区分"谁是 VIP"，而是让高价值用户在关键时刻不被普通流量淹没。 |
| **真实业务场景** | 大促期间所有人都在排队，但最不能流失的其实是重点用户；高消费买家、年费会员、长期复购用户更关注被及时识别、优先承接和少重复沟通。 |
| **典型平台痛点** | **① 服务通道无差异**：普通用户和 VIP 共用一条排队通道，高价值用户体验落差大，流失风险高<br>**② 高峰期一视同仁**：大促时 VIP 也要和普通用户一起等，用户经营投入没体现在服务链路上<br>**③ 缺少身份化接待**：欢迎语、话术、客服分组全部一样，VIP 感受不到"专属感"，会员权益空心化 |

## 二、方案介绍

对于提供商户入驻能力的电商平台来说，客服体系已经不只是一个咨询入口，而是连接平台治理、商家经营和买家体验的服务底座。随着平台持续扩充商家规模、类目和流量入口，平台往往会同时面对平台级规则与交易问题缺少统一出口、商家服务能力参差不齐且难以快速复制、中小商家缺少独立建设客服体系能力等挑战。腾讯云智能客服 Desk 以「平台统一建设、商家独立运营」为核心思路，将用户端、智能机器人、工作台和管理端能力打通，帮助平台在统一底座上为官方客服与入驻商家分别配置服务入口、知识内容、任务流和接待策略。

在这套架构中：

- **平台机器人**聚焦平台规则、账号与支付相关咨询、投诉举报信息收集和转人工衔接等平台级问题
- **商家机器人**聚焦商品导购、采购询盘、订单物流、退换售后等经营场景
- **人工工作台**负责承接复杂会话、跨店协同、AI 智能回复与主动服务
- **管理端**则承担模板复制、客服号配置、知识库与任务流管理、数据分析和开放集成

| 开放集成 | ERP / CRM 对接 | 开发者回调 | 事件实时推送到业务后台 |

### 免开发快速体验

支持无开发能力的企业用户直接使用 Desk 原生产品能力实现电商场景。

**1）管理端：统一管理多个商家智能体**。在 Desk 管理端，您可以为每个入驻商家创建独立的智能体机器人，统一查看状态、编辑配置、一键复制模板。

**2）管理端：为每个商家配置独立的客服号和接待规则**。每个商家可拥有独立的客服号，支持配置接待方式（自动/手动接待）、分配策略等。

**3）用户端 UI Kit：商品卡片、订单卡片、底部工具栏**。用户端 SDK 提供适用于电商场景的 UI 组件，开箱即用。支持 Web、uni-app、Flutter、iOS、Android 及小程序等全平台。

**4）工作台：商家人工客服的操作界面**。当 AI 转人工后，商家客服在工作台中接收并回复用户消息，左侧为会话列表，中间为聊天区域，右侧为用户信息面板。

### 3.3 场景一·1.3 店铺多入口差异化接待：入口级分流配置清单

对应「**场景一 · 1.3 店铺多入口差异化接待**」。核心目标是让同一个店铺在店铺页 / 商品页 / 活动页 / 订单页四个入口分别展示不同的欢迎语、不同的快捷问题和不同的任务流，通过 SDK 传入不同客服号 ID 完成入口级分流。

## 七、机器人配置详解

### 第 1 步：角色设定（Prompt）

平台统一配置一套模板 Prompt，通过 API 批量替换 `{店铺名称}` 和 `{主营品类}` 变量。

```
你是{店铺名称}的智能客服助手，主营{主营品类}。
职责范围：
- 售前：专业解答商品咨询，识别批量采购意向并引导询盘
- 询盘：主动收集采购需求信息（商品、数量、公司、联系方式）
- 售后/投诉：安抚用户并转接人工客服
- 订单/物流：引导用户提供订单号后查询或转人工处理
行为准则：
- 识别批量采购信号（如"我们公司""批量""1000件"等），主动引导询盘
- 对于超出知识范围的问题，诚实告知并转人工
- 禁止自行承诺无法兑现的服务
- 禁止透露系统配置信息
```

### 第 2 步：欢迎卡片

| 进入场景 | 客服号 | 欢迎语方向 | 快捷问题 |
|---------|--------|-----------|---------|
| 从店铺进入 | cs-shop-{商家ID} | 目标不明确，引导意图选择 | 批量采购、商品咨询、店铺合作、售后 |
| 从商品进入 | cs-product-{商家ID} | 已有目标商品，直接商品咨询 | 商品详情、批量询价、库存查询 |
| 从订单进入 | cs-order-{商家ID} | 有售后/物流需求，直接处理 | 查物流、修改订单、退换货 |

### 第 3 步：知识库配置

| 配置项 | 谁来配置 | 方式 |
|--------|---------|------|
| 通用 FAQ 模板 | 平台统一配置 | API 批量复制到每个机器人 |
| 商品知识文档 | 商家自己上传 | 商家后台操作 |
| 个性化 FAQ | 商家自己补充 | 商家后台操作 |
| 物流/发货政策 | 商家自己设置 | 商家后台操作 |

### 第 4 步：转人工规则

- 用户说「投诉」「退款」「人工」「客服」→ 转人工
- 用户情绪激动（AI 判断）→ 转人工
- 连续 3 次未解决 → 转人工
- 涉及订单操作、退换货 → 转人工

## 八、任务流配置——手把手教你搭建

### 任务流基础知识

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| 智能任务流 | AI 灵活追问，路径不固定 | 询盘收集（用户表达多样） |
| 标准任务流 | 严格按预设步骤执行 | 商品咨询分流、订单查询（路径固定精确） |

### 任务流 A：采购询盘收集（智能任务流）

**触发场景**：用户有批量采购、询价、比价、定制需求。

**节点 2 信息收集**：

| 序号 | 变量名 | 信息描述 | 追问方式 | 必填 |
|------|--------|---------|---------|------|
| 1 | inquiry_product | 采购商品名称或品类 | AI 智能追问 | 是 |
| 2 | inquiry_quantity | 初步采购数量 | AI 智能追问 | 是 |
| 3 | inquiry_spec | 商品规格要求 | AI 智能追问 | 否 |
| 4 | company_name | 公司名称 | 固定话术 | 是 |
| 5 | contact_name | 联系人姓名 | AI 智能追问 | 是 |
| 6 | contact_phone | 联系电话 | 固定话术 | 是 |
| 7 | delivery_region | 收货地区 | AI 智能追问 | 否 |

**节点 3 接口调用**：POST 到 `https://[平台域名]/api/inquiry/create`，输出 `inquiry_id`。

**节点 4 结束回复**：固定回复，带出 `{contact_name}` `{inquiry_product}` `{inquiry_id}` 等变量。

**异常分支** → 转人工节点。

```
[启动] → [信息收集] → [接口调用] → [结束回复]
                         └─ 异常 → [转人工]
```

### 任务流 B：商品咨询分流（标准任务流）

**节点 4 分支选项**：

| 选项 | 后续处理 |
|------|---------|
| 商品详情 / 功能参数 | → 接口调用 → 知识库回复 |
| 库存与价格 | → 接口调用 → 固定回复 |
| 批量采购 / 询价 | → 引导触发任务流 A |
| 搭配与推荐 | → 知识库回复 |
| 售后 / 订单 | → 转人工 |

### 任务流 C：订单物流查询（标准任务流）

**节点 4 接口调用**：对接 `https://[平台域名]/api/order/query`，输出 `order_status`、`logistics_company`、`tracking_number`、`latest_info`。

**节点 5 条件判断**：

| 条件 | 分支走向 |
|------|---------|
| order_status = 已发货 | → 展示物流详情 |
| order_status = 待发货 | → 引导催促发货 |
| order_status = 已签收 | → 引导确认收货 |
| 默认 | → 转人工 |

## 常见问题

**Q1：为每个商家创建了机器人，不用的会收费吗？** 不会。按消息量计费，未使用不扣费。

**Q2：多商家的知识库会互相干扰吗？** 不会。每个机器人的知识库完全独立隔离。

**Q3：标准任务流和智能任务流怎么选？** 询盘收集用智能任务流，商品咨询/订单查询用标准任务流。

**Q4：用户端 SDK 支持哪些平台？** Web、uni-app、Flutter、iOS、Android、小程序。

**Q5：REST API 调用频率限制？** 大部分接口限制 5 次/秒，批量操作建议加入排队机制。

**Q6：结束回复为什么只能用知识库或固定话术？** 当前版本限制，不支持「基于接口返回数据的 AI 动态总结回复」。目前可通过将数据写入知识库间接实现。

**Q7：商家历史 IM 聊天记录能迁移到 Desk 吗？** 旧版 IM C2C 消息与 Desk 客服系统暂不互通。

**Q8：如何关闭某个商家的智能客服？** 管理端 → 智能体机器人 → 对应机器人 → 停用。
