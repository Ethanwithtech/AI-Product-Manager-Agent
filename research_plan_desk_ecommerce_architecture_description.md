## 研究计划：Desk 电商平台能力架构方案描述

### 1. 需求拆解
用户要写的是“腾讯云智能客服 Desk · 电商平台能力架构”的方案描述，副标题为“场景驱动 · 每个业务场景对应具体的产品能力”。核心不是技术实现，而是面向平台型电商运营方，说明为什么需要这套架构、平台方会遇到哪些服务治理问题，以及 Desk 如何用平台机器人、商家机器人、工作台、管理端去承接这些问题。

### 2. 查询类型判断
这是一个**straightforward query**。目标明确，重点是形成一段准确、可落地、平台视角的方案描述。需要少量外部资料与内部能力边界核实即可完成，不需要大规模宽度研究。

### 3. 研究方法对比
方法 A：仅依据仓库现有 HTML 文案改写，速度快，但容易保留不准确能力表述。
方法 B：结合仓库文案 + Desk 能力边界知识库，保证能力名称和边界准确。
方法 C：结合仓库文案 + Desk 知识库 + 公开电商平台资料 + 微信公众号文章，既保证能力准确，也让平台方视角更自然。

最终采用**方法 C**。

### 4. 信息检索计划
#### 4.1 仓库内信息
- 重点文件：`desk-architecture-flow.html`、`desk-scene-table.html`、`desk-ecommerce-demo.html`
- 目标：提炼当前方案结构、已有场景名称、平台与商家职责划分。

#### 4.2 知识库检索
- 知识库：`Desk知识库`、`TCCC知识库`
- 目标：核实 Desk 是否支持投诉工单、跨商家调解、平台侧退款、优先排队、会话接管、用户画像、身份核验等能力，避免方案描述越界。

#### 4.3 微信公众号文章检索（必须带时间参数）
- Skill：`wechat-article-search`
- 时间范围：近 365 天
- 关键词：
  - `电商平台 平台客服 商家客服 职责边界`
  - `多商家 平台统一管理 商家独立运营`
  - `电商平台 会员服务 高价值用户 服务运营`
- 目标：吸收更贴近中文平台业务语境的表达方式。

#### 4.4 Web 检索
- 目标：补充公开平台资料、行业观点和职责边界表达。
- 重点信息：平台规则咨询、平台服务中心、招商入驻、会员服务、平台客服与商家客服分工。

### 5. 产出形式
1. 形成一版可直接放入方案/页面的正式描述；
2. 用平台方视角描述平台统一底座、多商家独立运营、平台级问题承接、工作台协同和管理端治理；
3. 避免出现已确认超出 Desk 能力边界的表述；
4. 如合适，将文案同步写入 `desk-architecture-flow.html`。
